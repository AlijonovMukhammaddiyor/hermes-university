# RFC-002 — University Lifecycle (the full, real-university model)

Status: **HISTORICAL** — canonical description is [`ARCHITECTURE.md`](../ARCHITECTURE.md). Date: 2026-07-07
> Superseded where it disagrees with later RFCs: the old hardcoded interview-readiness degree
> name is now the generic, profile-driven `credential_name` (RFC-005). Lifecycle + rendering stand.

## 0. Why
RFC-001 gave us a working daily loop. This RFC makes the system behave and *read* like a real
university across its whole lifecycle, and makes the curriculum **visible** (WebUI + Obsidian). Two
non-negotiables carry over: **numbers are computed by code, not the model**, and **no outcome without
a proof**. New non-negotiable: **everything a student would "see" is a rendered document generated
deterministically from the canonical data** — never hand-typed, never LLM-invented.

## 1. The lifecycle (the organizing spine)
```
Catalog ─▶ Registration (prereqs, credit-load, add/drop) ─▶ Syllabus ─▶ Academic Calendar
   ─▶ Instruction (daily, concept-first) ─▶ Assessment (weighted policy: hw/quiz/exam/midterm/finals)
   ─▶ Grading → Transcript → GPA ─▶ Standing (probation/good/honors + consequences)
   ─▶ Progression (unit prereq gates) ─▶ Promotion (semester finals ≥B) ─▶ Graduation (degree + diploma)
        └─ throughout: Advising / Degree-audit · Personalization (Learner Model) · Retention (SRS)
```

## 2. Canonical data — additions to RFC-001

### 2.1 Course module (`course.yaml`) — add a grading policy
```yaml
grade_weights:            # fractions that sum to 1.0 — the course's grading policy
  hw: 0.30
  quiz: 0.20
  exam: 0.20             # biweekly unit exams
  midterm: 0.10
  finals: 0.20
```
Every course declares this; the engine (not the LLM) turns grades into a course GPA.

### 2.2 Grade record — drop the LLM-chosen weight
`GradeRecord` no longer carries `credits_weight`. Weight is derived: a record's contribution =
`course.grade_weights[kind]` (split across that kind's records) × `course.credits`. The grader only
supplies the **band** + evidence; the engine computes everything downstream.

### 2.3 Grading math (deterministic, real-university)
- **course GPA** = Σ over kinds `weight[kind] × mean(grade_points of that kind's records)`, on a 0–4
  scale (kinds with no records yet are renormalized out).
- **semester GPA** = credit-weighted mean of the course GPAs of that semester's enrolled courses.
- **cumulative GPA** = credit-weighted mean across all graded courses.
- Standing from cumulative: `<2.5 probation · ≥3.7 honors · else good`.

### 2.4 State (`state.json`) additions
```jsonc
"enrollment": { "credit_cap": 14, "records": [ {code, enrolled_on, dropped_on?} ] },
"hold": null,                 // e.g. "probation" — blocks new material until cleared
"degree": { "name": "<credential name>",
            "requirement": "pass finals of both semesters (>=B)", "awarded_on": null }
```

### 2.5 Learner Model — persisted
`records/learner_model.json` is now written (not just recomputed) each audit: topics
(proficiency/ceiling/misconceptions), routine (best hours, adherence), pace. Per-card FSRS review
state stays in Anki (phone); the engine owns mastery/proficiency. The unused `fsrs` field is removed.

## 3. The rendering layer — the *visible* university (P0, the must-have)
A deterministic renderer (`engine/docs.py`) regenerates human-readable documents into the vault on
every state change, so the **WebUI file browser** and **Obsidian** show a real university, and the
agent can render any of them in chat on request:

| Document | Path | Content |
|---|---|---|
| **Catalog** | `Catalog.md` | every course: title, credits, prereqs, "what you'll be able to do", grading policy |
| **Syllabus** | `Courses/<C>/Syllabus.md` | outcomes by unit, the week-by-week schedule, grading policy, assessments, sources |
| **Transcript** | `Registrar/Transcript.md` | per-course units + bands + course GPA; semester + cumulative GPA; standing |
| **Schedule** | `Registrar/Schedule.md` | academic calendar: term start/end, add-drop, midterm/finals dates |
| **Degree progress** | `Registrar/DegreeProgress.md` | % of required outcomes mastered, credits earned, GPA, what's left, projected graduation |

Rule: these are **read-only projections** — the engine writes them; humans read them. Regenerated by
`hu-engine render-docs --vault … --courses …`, called at the end of assign/audit/enroll/promote.

## 4. Registration integrity (P0)
`enroll` now enforces, like a registrar:
- **Prerequisites**: every code in `course.prerequisites` must be **completed** (passed finals /
  degree-credit) — else refuse with a clear reason.
- **Credit load**: sum of enrolled active-course credits ≤ `enrollment.credit_cap` (default 14).
- **Hold**: if `state.hold` is set (e.g. probation), refuse new enrollment until cleared.
- Records an enrollment entry; `drop` records the drop.

## 5. Academic calendar (P1)
Derived from `program.started_on` + `weeks_per_semester`: term-1 start, add-drop deadline (+1wk),
midterm (wk6), finals (wk12), term-2 start (wk12), term-2 midterm/finals, projected graduation.
Rendered in `Schedule.md`; the assign uses it to know today's week + upcoming deadlines.

## 6. Standing consequences (P1)
- **Probation** (cum GPA <2.5): set `hold="probation"` → assign runs a **remediation week** (revisit
  weak/failed outcomes, no new units) and enrollment is blocked until GPA recovers.
- **Honors** (≥3.7): difficulty ramps faster / harder tiers unlock earlier (via the Learner Model).
- **Dean's-list / streak milestones**: celebrated in the digest (copy only).

## 7. Progression, promotion, graduation, degree (P1)
- Unit gates unchanged (prereq DAG + exit gate).
- Promotion unchanged (`promote`, S1 finals ≥B → S2).
- **Graduation**: passing S2 finals (≥B) awards the degree → `degree.awarded_on` set, a
  `Registrar/Diploma.md` rendered, a completion record in `history`. Degree audit shows credits +
  cumulative GPA at award.

## 8. Advising / degree-audit (P1)
`DegreeProgress.md` (rendered) + interactive Registrar verbs (`what's left`, `am I on track`) answer
the real-university advising questions from the deterministic data — never guessed.

## 8b. Messaging & content delivery — Telegram-native (P0/P1)
Telegram is the student's live campus. Hermes runs python-telegram-bot, so we use its real features
(grounded in what the gateway exposes):
- **Voice** — the message is a coach texting: short, warm, richly but lightly formatted (bold key
  items, emoji as signposts, never a report). Digest voice from RFC-001 stands; hide all internals.
- **Suggested quick-actions** — every digest ends with a compact reply menu the learner can tap-type,
  e.g. `Reply: done · reschedule · explain · status`. (Conversational "buttons"; inline-keyboard
  markup is a later enhancement if the gateway exposes reply_markup to skills.)
- **Voice answers in** — quizzes, Feynman teach-backs, and behavioral mocks accept a **voice memo**
  (auto-transcribed) as the answer — the natural way to practice speaking an interview answer.
- **Documents & media** — deliver artifacts as files via `hermes send -f`: the **Syllabus**, a
  **weekly progress card**, a **system-design diagram** image, the **Transcript** — not walls of text.
- **Topics/threads** — route by kind into Telegram topics for a tidy campus: `#daily` (assign/audit),
  `#exams`, `#announcements` (promotions, standing changes, milestones). Uses the cron thread id.
- **Content management principle** — the **canonical content lives in the vault** (lessons, syllabi,
  notes — viewable in WebUI/Obsidian); Telegram carries a **concise, actionable summary + the
  action menu + any file attachment**, and **Anki carries the spaced-repetition cards to the phone**.
  One source (vault), three surfaces (WebUI/Obsidian to read, Telegram to act, Anki to retain).
- **Cadence discipline** — one morning assign, one night audit, weekly assessment, event-driven
  announcements (promotion/standing/graduation). No chatter. Right message, right moment.

## 9. Coherence & invariants
- Engine owns all numbers + writes all rendered docs; skills/crons call the engine and read the docs.
- Every rendered doc is reproducible from `state.json` + `grades.jsonl` + `courses/` — delete them
  all and `render-docs` recreates them.
- Grade weights come from the course policy, never the model.

## 10. Build order (each tested before the next)
**P0:** (1) rendering layer + `render-docs` CLI; (2) deterministic grade weights (Course.grade_weights
+ gradebook rewrite + drop record weight + update course.yamls + grade CLI); (3) enrollment integrity
(prereq + credit-load + records). **P1:** (4) academic calendar + Schedule; (5) degree audit +
graduate/diploma; (6) standing consequences (hold); (7) learner-model persistence. **Then:** wire
skills/crons (regen docs on every state change; enroll enforces; deterministic weights), deploy,
verify a full simulated term (enroll → assign → grade → transcript → midterm → finals → promote →
graduate) end-to-end.
